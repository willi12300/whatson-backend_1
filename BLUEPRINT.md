## 1. Recommended Tech Stack

| Layer | Choice | Why |
|---|---|---|
| Runtime | **Node.js 20 LTS** | Fast I/O, massive ecosystem, your frontend is JS already |
| Framework | **Fastify** | 2× faster than Express, built-in schema validation, good plugin system |
| Database | **PostgreSQL 16 + PostGIS** | Best-in-class geospatial, JSON support, free |
| ORM | **Prisma** | Auto-generates types, migrations handled cleanly, great DX |
| Cache | **Redis 7** | Venue cache, rate-limit counters, dedup bloom filter |
| Job queue | **BullMQ** (on Redis) | Scheduled ingestion jobs, retry logic built in |
| Storage | **Cloudflare R2** | S3-compatible, free egress, cheap. Store proxied venue photos here |
| Auth | **Supabase Auth** (or Firebase) | OAuth (Apple/Google/phone) solved in hours, not weeks |
| Hosting (start) | **Railway.app** | Postgres + Redis + Node on one platform, free tier, git-push deploy |
| Local dev | **Docker Compose** | Postgres + Redis + PostGIS spun up locally in one command |

## 3. Database Schema (Prisma)

See `prisma/schema.prisma` — key design decisions:

- **Venue** has a `canonical = true` flag: after dedup, one record wins and duplicates point to it via `canonical_id`
- **geo** is a PostGIS `geography(Point, 4326)` — enables fast `ST_DWithin` radius queries
- **source_refs** is a JSON array — one venue can hold its Google, Foursquare, and OSM IDs in one place
- **Event** → **Venue** is many-to-one after matching; `matched_confidence` records how sure we are

Fetch once, save to R2, serve from there forever.

## 5. Data Ingestion Jobs

### Job: `ingest-venues` (runs weekly, Sunday 2am)
```
1. Run Google Places grid scan (12 cells × 6 types)
2. Run Foursquare scan
3. Run OSM scan
4. For each source result: normalise → dedup → upsert
5. Fetch & cache photos for new venues
6. Log: X new, Y updated, Z duplicates merged
```

### Job: `ingest-events` (runs every 4 hours)
```
1. Fetch Skiddle events (next 30 days)
2. Fetch Eventbrite events (next 30 days)
3. Fetch Ticketmaster events (next 30 days)
4. Dedup events within each source
5. Match each event to a venue record
6. Upsert events (update if already exists, insert if new)
7. Expire events past their end_time
```

### Job: `expire-content` (runs every 15 minutes)
```
1. SET offers.status = 'ended' WHERE valid_until < NOW()
2. SET posts.expired = true WHERE expires_at < NOW()
3. Refresh Redis venue cache for affected venues
```

## 6. Deduplication Logic

### Venue deduplication (run after each source ingest)

Uses a **3-signal scoring system**. If score ≥ 2, records are duplicates:

```
Signal 1 — Name similarity (1 point if score ≥ 0.85)
  Normalise: lowercase, remove "the/a/an", strip punctuation, trim
  Algorithm: Jaro-Winkler distance (good for pub names with apostrophes)

Signal 2 — Proximity (1 point if distance < 80 metres)
  Use PostGIS: ST_DWithin(geo, other_geo, 80)

Signal 3 — Cross-source ID match (1 point — strong signal)
  Google place_id in Foursquare linked_ids, or
  phone number match, or website domain match
```

When duplicate found:
1. **Master record** = prefer Google > Foursquare > OSM (best data quality)
2. Copy any missing fields from secondary to master (OSM may have opening hours Google lacks)
3. Mark secondary `canonical = false`, set `canonical_id = master.id`
4. Merge `source_refs` arrays

### Event deduplication

```
Two events are duplicates if:
  - Same venue_id AND
  - Name similarity ≥ 0.80 AND
  - Start time within 2 hours of each other
  
Master: prefer Ticketmaster > Skiddle > Eventbrite (ticket link quality)
```

## 7. Event-to-Venue Matching Logic

Events from Skiddle/Eventbrite/Ticketmaster include a venue name + address but no database ID. Matching logic:

```typescript
async function matchEventToVenue(event: RawEvent): Promise<string | null> {
  // Step 1: exact external ID match (Ticketmaster includes a venue ID)
  if (event.externalVenueId) {
    const v = await db.venue.findFirst({
      where: { source_refs: { path: '$.ticketmaster_id', equals: event.externalVenueId } }
    })
    if (v) return v.id
  }

  // Step 2: name + proximity (within 200m, name similarity ≥ 0.75)
  const candidates = await db.$queryRaw`
    SELECT id, name,
      ST_Distance(geo::geography, ST_MakePoint(${event.lng}, ${event.lat})::geography) as dist
    FROM venues
    WHERE ST_DWithin(geo::geography, ST_MakePoint(${event.lng}, ${event.lat})::geography, 200)
    AND canonical = true
    ORDER BY dist ASC
    LIMIT 10
  `
  for (const c of candidates) {
    const sim = jaroWinkler(normalise(c.name), normalise(event.venueName))
    if (sim >= 0.75) return c.id  // take best proximity + name match
  }

  // Step 3: name-only fuzzy search (no coordinates in event)
  const byName = await db.$queryRaw`
    SELECT id FROM venues
    WHERE similarity(name, ${normalise(event.venueName)}) > 0.6
    AND city = 'Liverpool'
    ORDER BY similarity(name, ${normalise(event.venueName)}) DESC
    LIMIT 1
  `  // requires pg_trgm extension
  if (byName.length) return byName[0].id

  // Step 4: no match — create a stub venue from event data
  return await createStubVenue(event)
}
```

## 9. Cost-Saving / Caching Strategy

### The golden rule: **never hit an external API on a user request**
All external APIs run in background jobs. The frontend only ever talks to YOUR API.

### Venue data (slow-changing)
```
Redis key: venue:{id}  →  full venue JSON
TTL: 24 hours
Warm: after each ingestion run
Invalidate: when venue is updated by owner

Redis key: venues:nearby:{lat_rounded}:{lng_rounded}:{radius}
TTL: 15 minutes (changes as new venues get ingested)
This key handles 90% of map/feed requests
```

### Event data (faster-changing)
```
Redis key: events:venue:{venue_id}  →  upcoming events array
TTL: 4 hours (matches ingestion frequency)

Redis key: events:nearby:{lat_rounded}:{lng_rounded}
TTL: 30 minutes
```

### Photo proxying (the big cost saver)
Google Places photo URLs expire and are metered. Solution:
1. After venue seed, fetch each photo ONCE via the Google URL
2. Save to Cloudflare R2 (free egress, ~$0.015/GB storage)
3. Serve forever from R2 — Google never gets another request for that photo
4. R2 URL stored in `venues.photos[]`

### Coordinate rounding for cache keys
```typescript
// Round to 3 decimal places (~111m precision) for cache key
const cacheKey = `venues:nearby:${lat.toFixed(3)}:${lng.toFixed(3)}:${radius}`
// Different users 50m apart share the same cache → massive hit rate
```

### Google Places budget estimate for Liverpool MVP
- Initial seed: ~60 requests × $0.032 = **$1.92** (one-time)
- Weekly refresh: ~60 requests × $0.032 = **$1.92/week**
- Photo fetch: ~500 venues × 1 photo × $0.007 = **$3.50** (one-time, then R2 forever)
- **Total: ~$10 first month, ~$2/month thereafter**
